# Input Quality Measurement Framework for RESTest
Date: 2026-04-26 (rev. 2 — scope corrected to black-box microservice testing)
Companion: `microservice-input-quality-research.md` (field survey + bibliography)

## Scope

This framework measures the quality of **inputs** generated by RESTest's Smart-Fetch and LLM pipelines, **for testing microservice REST APIs in black-box mode**. It is constrained to:

1. **Input-intrinsic only** — every metric is computable from the generated value plus static metadata (OpenAPI spec, IDL, prompt-stated constraints, fault-type label, smart-fetch provenance, Jaeger trace export). **No SUT mutation, no oracle, no source code, no JaCoCo.**
2. **Black-box microservice context** — the SUT is the TrainTicket benchmark (41 services across multiple JVMs/containers). Source-code instrumentation is not available. The only runtime evidence is HTTP request/response and Jaeger distributed traces — both produced *as a side effect* of running tests, not as a probe.
3. **Microservice-aware** — three of the ten metrics target distributed-system-specific concerns (cross-service ID grounding, trace-resolvability, producer-consumer chain integrity) that monolithic-API frameworks under-specify.

## Out of scope (and why)

The following were in the previous revision and are **moved out**. They are not input-quality metrics: they measure how the *generator + oracle + SUT* combine, not the inputs themselves. They belong in a separate **tool-effectiveness framework** (to be written when the user requests it).

| Metric | Why it is not input-level |
|---|---|
| Mutation score | Requires running mutated SUTs; measures suite distinguishing power [13, 21]. |
| Real fault detection rate | Requires ground-truth bug list and oracles; measures tool+oracle+SUT [5, 8, 21]. |
| Injected fault detection rate | Needs the injection harness and the oracle. The user explicitly flagged this — **injected faults are designed for whole-tool performance measurement, not input quality.** [21] |
| Code coverage (line / branch / method) | Black-box impossible. Even with white-box access, it measures *SUT execution given the input*, not the input itself [3, 5, 21]. |
| Status-code coverage | Restats includes it [4] but it is *server output*, not input. The status code is the SUT's reaction; two valid inputs both yield 200 [4]. |
| Negative-test rejection rate (4xx fraction) | *Server response*, so tool/oracle-level. The input-level analogue is **D10 NIFP** below — "did we generate inputs that *should* be rejected per the schema?" — purely input-side. |
| 5xx rate / silent-acceptance rate | Server-output behavior. Tool-level. |

A pointer to where these belong: `debug/inputs/tool-effectiveness-framework.md` (not yet written).

## The Ten Input-Only Metrics

Organised in five families. Each metric is operationally defined, citable, and computable from artefacts RESTest already produces.

| # | Metric | Family | Microservice-specific? |
|---|---|---|---|
| **D1** | Schema Conformance Rate (SCR) | Validity | No (universal) |
| **D2** | Inter-Parameter-Dependency Satisfaction Rate (IPD-SR) | Validity | No |
| **D3** | LLM Hallucination Rate (LHR) | Validity (LLM-only) | No |
| **D4** | **Smart-Fetch Hit Rate (SFHR)** | **Microservice grounding** | **Yes** |
| **D5** | **ID-Resolvability Rate (IDR)** | **Microservice grounding** | **Yes** |
| **D6** | **Chain Resolution Rate (CRR)** | **Microservice grounding** | **Yes** |
| **D7** | Realism Score (ARTE-style) | Realism | No |
| **D8** | Pool Shannon Entropy + Simpson Index | Diversity | No |
| **D9** | Equivalence-Partition Coverage (EPC) | Diversity | No |
| **D10** | Negative-Input Fault-Type Purity (NIFP) | Negative-input adversariness | No |

D4–D6 are the centerpiece additions for microservices. They measure the property that distinguishes a deployable test from a stochastic stub: **does the input refer to a real entity that the upstream service in the data-dependency chain would have actually produced?** This concept is articulated in RESTler's producer-consumer model [1], formalised as a value-source weight in ARAT-RL [11], and made trace-explicit by OpenTelemetry trace-based testing [10].

## Family 1 — Validity (D1, D2, D3)

### D1. Schema Conformance Rate (SCR)

**Definition.** Fraction of generated input values that satisfy *every* applicable JSON-Schema/OpenAPI constraint of the parameter (type, format, regex, enum, min/max, minLength/maxLength, required).

**Formula.**
```
SCR(P) = |{ v in V_pos(P) : conforms(v, schema(P)) = true }| / |V_pos(P)|
```
where `V_pos(P)` is the multiset of *positive* (non-fault-injected) values generated for parameter `P` and `conforms()` is a deterministic JSON-Schema validator (e.g., `swagger-parser`, `jsonschema`).

**Why input-level.** Computing SCR requires only the value and the schema. No SUT call.

**Citations.** Restats [4] proposes parameter and parameter-value coverage on the same artefact set. Schemathesis [22] is the canonical client-side schema validator for OpenAPI. Atlidakis et al. (RESTler) [1] use schema-conformant request rate as their primary signal. Mirabella et al. [37] predict input validity via deep learning on the same schema-vs-value comparison. RESTGPT [14] and LlamaRestTest [16] both report variants of this rate as their headline input-level number.

**How to compute in RESTest.** The generated test files (`target/test-classes/.../Flow_Scenario_*.java`) record each request's parameters. Re-emit them as `(parameter, value)` pairs and feed each into `swagger-parser` against the corresponding `oas-spec.yaml`. SCR per parameter, per operation, per service, and overall.

**Reasonable thresholds.** RESTGPT reports 73% valid-input rate vs. ARTE's 17% on a benchmark [14]; LlamaRestTest improves further [16]. **SCR ≥ 0.90 per parameter** is a reasonable target for the positive path; lower values flag a generator regression.

---

### D2. Inter-Parameter-Dependency Satisfaction Rate (IPD-SR)

**Definition.** Fraction of generated request payloads that satisfy all IDL constraints (`OnlyOne`, `AllOrNone`, `Or`, `Requires`, `ZeroOrOne`).

**Formula.**
```
IPD-SR(O) = |{ payload p for operation O : every idl in IDL(O) is satisfied by p }| / |payloads(O)|
```

**Why input-level.** IDL constraints are static metadata on the schema. Satisfaction is decidable from the payload alone via the IDL parser RESTest already ships.

**Citations.** Martin-Lopez et al. [24, 28] introduce IDL and the Choco-solver-based satisfaction check. RESTGPT [14] reports 9/17 vs. base Llama3-8B's 2/17 on IPD case detection. LlamaRestTest [16] fine-tunes a dedicated IPD model (`LlamaREST-IPD`) and reports it as one of two headline contributions.

**How to compute.** Instrument `MultiServiceTestCaseGenerator` to log a boolean `idl_satisfied` per generated request. Aggregate per operation.

---

### D3. LLM Hallucination Rate (LHR)

**Definition.** For LLM-emitted values only: fraction that violate at least one constraint *that the prompt explicitly stated*.

**Formula.**
```
LHR = |{ v in V_LLM : there exists c in Constraints(prompt(v)),  v violates c }| / |V_LLM|
```

**Why input-level.** Both the prompt's stated constraints and the LLM's response are recorded in `logs/llm-communications/*.log`. No SUT involvement.

**Citations.** Vectara Hallucination Leaderboard [18] and HalluLens [15] formalise the methodology of comparing model output to a known constraint. RESTGPT [14] reports a 73% valid-input rate as a hallucination proxy. LlamaRestTest [16] reports 72.44% valid-input-generation rate.

**How to compute.** Parse LLM logs for prompt blocks, regex out stated constraints (`Enum: …`, `Pattern: …`, `Min/Max: …`, `Format: …`), validate each emitted value. Track LHR per model (`Model Name:` header line) so a model regression is visible at swap time.

**Reasonable thresholds.** 5–15 % is typical for current open-weight 8B–14B models on REST domains [16]. > 30 % suggests the prompt is not being read.

## Family 2 — Microservice grounding (D4, D5, D6) — *the centerpiece for microservices*

### D4. Smart-Fetch Hit Rate (SFHR)

**Definition.** For ID-typed parameters (UUID, accountId, orderId, …), the fraction whose generated value came from a real upstream service response — not from LLM fabrication or random padding.

**Formula.**
```
SFHR = |{ v in V_ID : provenance(v) in {SMART_FETCH, TRACE_REPLAY, JIT_BINDING, DATA_PROVENANCE} }|
       /
       |V_ID|
```
where `V_ID` is the multiset of values used for ID-typed parameters across all generated tests, and `provenance` is the source recorded by RESTest's input pipeline (from `dataflow-map.md`: priorities 2-PROV, 3-JIT, 4 trace replay, 5 shared pool, 6 smart fetch).

**Why this is THE microservice metric.** In TrainTicket, an `accountId` flows through `ts-auth → ts-order → ts-travel → ts-ticket-purchase`. A fabricated `accountId` causes the chain to fail at hop 2; every downstream call is then exercised against a 4xx — the test runs but tells you nothing about deeper services. SFHR directly quantifies how much of the test population is actually exercising real cross-service flows.

**Citations.** RESTler [1] introduced the producer-consumer model — input X to operation B is high quality iff some operation A has been observed producing X. ARAT-RL [11] formalises this as a value-source weight in its Q-table: `random < default < example < request-history < response-history`. DeepREST [12] uses curiosity-driven RL to learn implicit input-output dependencies. Morest [6] models the API as a producer-consumer DAG and prefers paths along its edges. AutoRestTest [30] uses LLM agents over a semantic graph to extend the same idea.

**Why it is input-level (not tool-level).** Provenance is a property of *how the input was produced*, recorded at generation time. It does not depend on the SUT response, the oracle, or any fault injection.

**How to compute.** RESTest already tags each parameter resolution with a priority (1–7 in the dataflow map). Bucket counts:
- Numerator: priorities 2-PROV + 3-JIT + 4 + 6 (real upstream values).
- Denominator: every ID-typed parameter resolution.
- Exclude priority 5 (shared pool) only when the pool itself was filled from LLM/padding, otherwise count it.

Emit one CSV per run: `(parameter, scenario, root_index, provenance, value)`. SFHR aggregates this.

**Reasonable thresholds.** Multi-trace TrainTicket runs with smart-fetch enabled should reach **SFHR ≥ 0.6 for ID-typed parameters**. Below 0.3 means the smart-fetch system is effectively idle.

---

### D5. ID-Resolvability Rate (IDR)

**Definition.** A *stronger, trace-grounded* version of D4. For each ID-typed value generated, look it up in the **set of all values observed as outputs in any Jaeger span** during the test run. IDR is the fraction whose value appears in that ground-truth set.

**Formula.**
```
IDR = |{ v in V_ID :  v appears in Outputs(JaegerExport) }| / |V_ID|
```

**Why this is different from D4.** D4 records provenance *at generation time* — "the smart-fetcher said it got this from service X." D5 verifies provenance *at execution time* using the actual distributed trace. If the smart-fetch was stale (the value was created in a previous run and the SUT has since deleted it), D4 says "real" but D5 correctly reports "not resolvable in this run's traces." IDR is therefore the more rigorous metric; D4 is the cheap precondition.

**Citations.** Jaeger [23] is the canonical distributed-tracing platform for the TrainTicket benchmark [9, 33]. The OpenTelemetry trace-based-testing community [10] argues exactly this point — distributed traces are the ground truth for cross-service data flow. Train Ticket-specific papers [9, 17] use traces as the source of fact. Abdelfattah et al. [29] propose end-to-end coverage metrics for microservices that are similarly trace-grounded but on the operation side, not the value side; D5 fills the value-side gap.

**Why it is input-level.** The check is a set-membership test against an artefact (Jaeger export) the test run produces as a side effect. No oracle, no mutation, no source.

**How to compute.**
1. After the run, export traces from Jaeger (`logs/` or the configured Jaeger HTTP API).
2. Parse spans; for each span, extract output fields (response body keys, response headers, span tags ending in `Id`/`uuid`).
3. Build the global set `Outputs = union over all spans of outputs(span)`.
4. For each generated test, identify ID-typed inputs from the OAS; check membership.
5. IDR = positive count / total ID inputs.

**Reasonable thresholds.** IDR ≥ 0.5 indicates the smart-fetch subsystem is producing values that survive end-to-end. IDR < 0.2 indicates either Jaeger trace gaps (`http.response.body` missing) or massive smart-fetch staleness.

---

### D6. Chain Resolution Rate (CRR)

**Definition.** For multi-step test sequences, the fraction in which **every** consumed ID was produced by an earlier step *of the same sequence* (not just by some other test).

**Formula.**
```
CRR = |{ sequence S :
            for every step k in S, for every ID-typed input p in step k,
            there exists step j < k in S that produced value(p) }|
      /
      |sequences|
```

**Why this matters for microservices.** Microservice tests are workflows: login → search → create → use → cancel. A high SFHR can mask broken chains where the smart-fetcher is pulling IDs from outside the test sequence — fine for the immediate request but unrealistic for the workflow. CRR rewards tests that are self-contained chains (which are what production traffic actually looks like).

**Citations.** Morest [6], DeepREST [12], and ARAT-RL [11] all model this dependency. The Train Ticket benchmark [9, 33] explicitly emphasises stateful workflow testing. KAT [27] and AutoRestTest [30] use LLM agents to construct dependency-respecting chains; CRR is the natural quality measure for their output. The 2025 static-vs-dynamic constraint paper [32] frames this as the core open problem in modern REST testing.

**Why it is input-level.** The chain is a property of the generated test sequence and the OAS-declared (or registry-mined) dependency graph. No SUT call needed.

**How to compute.** For each generated `MultiServiceTestCase`:
1. Build the local producer set `P = union over k of step[k].outputs (declared producers in OAS / registry)`.
2. For each step `k`, every ID-typed input is *resolved* if some earlier step `j < k` produces a matching ID stem.
3. The sequence is *fully resolved* iff every ID input across all steps is resolved.
4. CRR = fully-resolved sequences / total sequences.

The data-flow map already documents `SemanticDependencyRegistry.findProducer` and `WorkflowStep.dataProvenance` — both feed this metric directly.

**Reasonable thresholds.** Multi-root TrainTicket scenarios after Phase-1/2 merging should hit CRR ≥ 0.7. Standalone 1-root scenarios trivially CRR = 1.

## Family 3 — Realism (D7)

### D7. Realism Score (ARTE-style)

**Definition.** For natural-language-typed parameters (city, station, country, person name), the fraction of values that resolve to a real entity in DBpedia / Wikidata / a domain corpus.

**Formula.**
```
Realism = |{ v in V_NLP : v matches an entity in KB }| / |V_NLP|
```

**Citation.** Alonso et al. (ARTE) [2, 20] is the canonical reference. ARTE itself reports 64.9 % realism on its target parameters and 57.3 % valid-call rate (vs. 26 % SAIGEN, 20 % random). RESTGPT [14] and LlamaRestTest [16] explicitly compare against ARTE on realism for NLP parameters.

**Why it is input-level.** KB membership is a static lookup against an external corpus (DBpedia SPARQL endpoint or local Wikidata dump).

**How to compute.** RESTest's ARTE module already implements this. Emit the hit/miss flag to the new validity CSV.

## Family 4 — Diversity (D8, D9)

### D8. Pool Shannon Entropy + Simpson Index

**Definitions.**
```
H(P) = - sum over uniq(V_P) of  f(v) * log_2 f(v)        (Shannon entropy)
S(P) = 1 - sum over uniq(V_P) of  f(v)^2                  (Simpson diversity)
```
where `f(v) = count(v) / |V_P|` over the parameter pool.

**Citations.** Shannon's original paper [25] is the source. Manès et al.'s fuzzing survey [19] establishes diversity-based metrics as input-quality proxies in the absence of coverage. Böhme's STADS theory [36] grounds species-discovery-style residual-risk arguments in black-box settings.

**Why it is input-level.** Distribution over generated values; no SUT.

**How to compute.** Hook the shared parameter pool dump (already serialised to disk for some runs). Plot `H_P(t)` over generation iterations — stagnation flags pool collapse (audit finding #18: rotation index leak).

**Reasonable thresholds.** For an n-element pool of distinct values drawn uniformly, `H = log_2 n`. Real pools should hit ≥ 80% of that ceiling; below 50% means duplication is dominating.

---

### D9. Equivalence-Partition Coverage (EPC)

**Definition.** Fraction of canonical equivalence classes for the parameter that contain at least one generated value. Classes are derived from the OAS schema:
- Numeric: `(-inf, min)`, `[min, mid)`, `[mid, max]`, `(max, +inf)`, `0`, `negative`, …
- String: `empty`, `< minLength`, `[minLength, maxLength]`, `> maxLength`, `format-conforming`, `format-violating`, …
- Enum: each declared enum value is its own class.

**Formula.**
```
EPC(P) = |{ classes(P) for which V_P contains at least 1 value }| / |classes(P)|
```

**Citations.** Kuhn et al. (NIST SP 800-142, "Practical Combinatorial Testing") [13] is the canonical reference for partition-based input coverage in black-box settings. Restats's parameter-value coverage [4] is a special case of EPC where the partitions are individual `enum` values.

**Why it is input-level.** Partitions are a function of the schema; coverage is a function of the pool. No SUT.

## Family 5 — Negative-Input Adversariness (D10)

### D10. Negative-Input Fault-Type Purity (NIFP)

**Definition.** For each negative input labelled with one of RESTest's 8 fault types, the fraction of inputs that **actually exhibit** that fault as a property of the value (not as an outcome of running it against the SUT).

**Per-type purity check.**

| Fault type | Purity check on the value |
|---|---|
| TYPE_MISMATCH | `actualType(v) ≠ schemaType(P)` |
| BOUNDARY_VIOLATION | `v < minimum or v > maximum` (numeric) OR `len(v) < minLength or len(v) > maxLength` (string) |
| REGEX_MISMATCH | `not regex(P).matches(v)` |
| OVERFLOW | numeric: `abs(v) > 2^31` (or > schema max). string: `len(v) > schema maxLength * 10` |
| EMPTY_INPUT | `v in {"", " ", "\t", "\n", [], {}}` AND parameter is required |
| NULL_INPUT | `v in {null, "null", "NULL"}` AND parameter is required |
| SPECIAL_CHARACTERS | regex match against an OWASP injection-pattern catalog [26] |
| SEMANTIC_MISMATCH | domain-specific (age = -5, future birthdate, etc.) — best-effort |

**Formula (per type T).**
```
NIFP(T) = |{ v labelled type T : purity_check_T(v) is true }| / |{ v labelled type T }|
```

**Why this matters.** Pipeline-bug-audit finding #6 (boolean TYPE_MISMATCH coerced to valid `Boolean.FALSE`) was a NIFP violation: the *label* said TYPE_MISMATCH, but the *value* was a perfectly valid boolean. NIFP catches that bug entirely from the input side — the SUT does not need to be involved.

**Why it is input-level.** Purity is a property of the value vs. the schema vs. the fault label. No SUT response, no oracle.

**Citations.** Restats's nominal-vs-error scenario classification [4] is the source of the fault-label discipline. NIST SP 800-142 [13] formalises partition-violation testing. OWASP API Security Top 10 [26] catalogs the SPECIAL_CHARACTERS injection patterns. The 2025 oracle-gap paper [21] explicitly separates "did the input contain the right kind of badness" (input quality) from "did the oracle catch it" (tool quality) — D10 is the former, the latter belongs to the tool-effectiveness doc.

**Reasonable thresholds.** Per fault type, NIFP ≥ 0.95 — the negative-input pool should be almost 100% pure for the labelled fault, otherwise the negative-test variants are silently bogus.

## Operational Protocol

A six-step protocol for measuring all ten metrics on a single RESTest run.

### Step 1 — Instrument once

Add an `InputQualityLogger` to RESTest that, for every parameter resolution, emits one CSV row:

```
run_id, scenario_id, root_idx, step_idx, operation, parameter, location,
schema_type, value, value_class, provenance, fault_label, idl_satisfied
```

`provenance` ∈ { `CONTEXT_OUTPUT`, `CONTEXT_REUSE`, `DATA_PROVENANCE`, `JIT_BINDING`, `TRACE_REPLAY`, `SHARED_POOL`, `SMART_FETCH`, `LLM`, `FALLBACK` }

`fault_label` ∈ { `POSITIVE`, `TYPE_MISMATCH`, `BOUNDARY_VIOLATION`, `REGEX_MISMATCH`, `OVERFLOW`, `EMPTY_INPUT`, `NULL_INPUT`, `SPECIAL_CHARACTERS`, `SEMANTIC_MISMATCH` }

This is the single artefact that drives all ten metrics. ~50 lines of code in `MultiServiceTestCaseGenerator.traverse()` and the negative-pool fire path.

### Step 2 — Mine the LLM communication log

Parse `logs/llm-communications/*.log` into pairs of `(prompt_constraints, response_values)` for D3 (LHR).

### Step 3 — Mine the Jaeger export

Export traces (the run already produces them under `logs/`) into a flat `(span_id, service, operation, output_field, output_value)` table. This is the ground-truth set for D5 (IDR).

### Step 4 — Run the metric calculator

A single Python notebook that consumes the three artefacts (Step 1 CSV, LLM log, Jaeger flat table) and produces:

- Per-parameter table: SCR, EPC, Realism, Shannon, Simpson, IDR.
- Per-operation table: IPD-SR, NIFP per fault type.
- Per-run scalar: SFHR, CRR, LHR, overall SCR.

### Step 5 — Calibrate and CI-gate

Set thresholds based on three pilot runs (small, medium, full TrainTicket). Recommended starting gates:

| Metric | Threshold | Reason |
|---|---|---|
| SCR (positive) | ≥ 0.90 | Below = generator regression |
| IPD-SR | ≥ 0.85 | Below = IDL extraction broken |
| LHR | ≤ 0.20 | Above = prompt not being read |
| SFHR | ≥ 0.60 | Below = smart-fetch idle |
| IDR | ≥ 0.40 | Below = Jaeger gaps or stale fetches |
| CRR | ≥ 0.50 | Below = chains broken |
| Realism | ≥ 0.50 | Below = ARTE regression |
| Shannon ratio | ≥ 0.80 | Below = pool collapse |
| EPC | ≥ 0.70 | Below = corner classes uncovered |
| NIFP (per type) | ≥ 0.95 | Below = negative tests are silently bogus |

### Step 6 — Track over time

A small JSON ledger per run; plot the ten KPIs over the last N runs. Flag any KPI that crosses its threshold.

## Why this scope is correct

Three reasons.

1. **Microservice systems are black-box from the test tool's perspective**, even when each service has its own source. Test tools cannot statically reach into 41 different containers' bytecode. Code coverage and mutation score are not feasible without invasive infrastructure that defeats the "deploy and test" model.

2. **Inputs and verdicts are conceptually different things**, and conflating them produces noisy metrics. Two scenarios:
   - High input quality, weak oracle → low fault detection, but inputs are good. A mixed-into-input-quality metric blames the inputs.
   - Low input quality, strict oracle → high fault detection driven by lucky inputs. A mixed metric praises the inputs.
   In both, the input-only metric (D1–D10) gives the truthful read; the conflated metric does not.

3. **Microservices need cross-service grounding metrics that monolithic-API frameworks omit.** The literature on microservice testing (Train Ticket [9, 33], OpenTelemetry trace-based testing [10], DeepREST [12]) consistently points to producer-consumer chains and trace-grounded ID resolvability as the dominant input concerns. D4–D6 fill that gap.

---

## What lives in the separate tool-effectiveness framework

The metrics removed from this document still matter. They go in a separate `tool-effectiveness-framework.md` (to be authored when the user requests). It will cover:

- Injected-fault detection rate (per the existing fault-detection harness).
- Real fault detection rate (when seeded-bug catalogues are available).
- Mutation score (when SUT mutation infrastructure is added).
- Restats-style status / response-type / path / operation coverage [4, 29].
- Status-code targeting effectiveness (LLM Status Code Discovery + Coverage Tracker).
- Soft-error oracle precision and recall.
- Test-case enhancer round-over-round improvement.

Each of those metrics is **end-to-end** (input + SUT + oracle + reporter) and answers a different question: *given high-quality inputs, does the rest of the tool do its job?* They depend on D1–D10 being healthy, but they are not D1–D10.
